✅ **基于文档内容判断相关性**（而非文件路径）

### 相关性判断策略

**改进前（不准确）**:
```python
def is_relevant(doc_path: str) -> bool:
    return any(keyword in doc_path.lower() for keyword in relevant_keywords)
```

**改进后（准确）**:
```python
def get_relevance_score(content: str, keywords: List[str]) -> int:
    """
    基于文档内容判断相关性
    0: 不相关, 1: 弱相关, 2: 中等相关, 3: 强相关
    """
    matched_count = sum(1 for kw in keywords if kw.lower() in content.lower())
    return min(matched_count, 3)
```

## 💡 核心发现

### 1. Python 的优势场景

✅ **语义理解能力强**
- rank-bm25 能更好地理解中文语境
- 对于"性能优化 缓存"这样的查询，能准确找到相关文档

✅ **排名质量高**
- MRR 显著高于 TypeScript（0.5556 vs 0.1528）
- 相关文档通常排在更靠前的位置

✅ **召回率高**
- 在大多数查询中，Python 的 Recall@K 更高
- 能找到更多相关文档

### 2. TypeScript 的优势场景

✅ **特定关键词匹配**
- 对于包含明确英文关键词的查询（如 "hybrid"）
- FTS5 的精确匹配有时更有效

✅ **性能稳定**
- 基于索引查询，不受文档数量影响
- 适合大规模知识库

### 3. 共同的问题

❌ **相关性判断依赖文件路径**
- 当前测试使用文件路径中的关键词判断相关性
- 不够准确，应该基于文档内容

❌ **中英文混合查询处理不佳**
- "JWT 认证 权限"查询两端都失败
- 需要改进分词和相关性判断逻辑

## 🎯 方案推荐

### 短期（当前）

**根据场景选择**:

#### 选择 Python (Qdrant + rank-bm25) 如果：
- ✅ 知识库规模小 (< 10,000 文档)
- ✅ **召回准确性是首要目标**
- ✅ 有足够的内存资源
- ✅ 查询频率不高

#### 选择 TypeScript (SQLite FTS5) 如果：
- ✅ 知识库规模大 (> 10,000 文档)
- ✅ **性能是首要目标**
- ✅ 内存资源有限
- ✅ 高并发查询

### 中期（1-3 个月）

**混合方案**:

```
用户查询
    ↓
路由层
    ├─→ 小规模/高精度需求 → Python (rank-bm25)
    └─→ 大规模/高性能需求 → TypeScript (FTS5)
```

**或者**:

```
两阶段搜索:
1. TypeScript FTS5 快速召回 Top-50
2. Python rank-bm25 精排 Top-10
```

### 长期（3-6 个月）

**引入专业搜索引擎**:

推荐：**Meilisearch** 或 **Elasticsearch**

优势:
- ✅ 召回准确性高（接近 rank-bm25）
- ✅ 性能好（接近 FTS5）
- ✅ 内存占用适中
- ✅ 功能丰富（模糊搜索、拼写纠正等）
