# 代码规范

## 1. 缩进与格式
- **缩进**：统一使用 **4个空格** 进行缩进，禁止使用 Tab 字符。
- **行宽**：每行代码建议不超过 120 个字符，超过时请合理换行。
- **空行**：
  - 函数/类定义之间保留一个空行。
  - 逻辑块之间可根据需要保留一个空行以增强可读性。
  - 文件末尾保留一个空行。

## 2. 注释规范
- **语言**：注释应使用中文，确保清晰、准确、简洁。
- **内容**：
  - 公共 API、复杂逻辑、算法实现必须添加注释。
  - 注释应解释“为什么”这样做，而不仅仅是“做了什么”。
- **禁止项**：
  - **严禁在注释中使用 Emoji 表情包**（如 ✅, ❌, 🚀, 💡 等）。
  - 避免无意义的注释（如 `i++ // i加1`）。

## 3. 命名规范
- **变量/函数**：使用 `camelCase`（小驼峰命名法），例如 `userName`, `getData()`。
- **类/接口/类型**：使用 `PascalCase`（大驼峰命名法），例如 `UserService`, `UserInfo`。
- **常量**：使用 `UPPER_SNAKE_CASE`（全大写加下划线），例如 `MAX_RETRY_COUNT`。
- **私有成员**：以下划线 `_` 开头，例如 `_internalState`。

## 4. TypeScript 特定规范
- **类型定义**：
  - 优先使用 `interface` 定义对象结构，使用 `type` 定义联合类型或工具类型。
  - 避免使用 `any`，除非确实无法确定类型，此时应添加注释说明原因。
  - 函数参数和返回值必须显式声明类型。
- **导入顺序**：
  1. 第三方库导入
  2. 项目内部绝对路径导入
  3. 相对路径导入
  每组之间保留一个空行。

## 5. 其他最佳实践
- **错误处理**：必须对异步操作进行适当的错误捕获和处理，避免静默失败。
- **不可变性**：优先使用 `const`，仅在需要重新赋值时使用 `let`，禁止使用 `var`。
- **函数设计**：保持函数单一职责，函数长度不宜过长（建议不超过 50 行）。
